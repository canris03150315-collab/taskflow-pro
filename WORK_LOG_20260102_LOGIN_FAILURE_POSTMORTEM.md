## 🔍 根本原因分析

### 問題 1：資料目錄損壞

**發現過程**:
1. 最初以為是 `auth.js` 代碼問題
2. 檢查並修復 `auth.js` 的 exports 位置
3. 回溯到多個舊版本 Docker 映像，問題依然存在
4. 用戶提供關鍵信息：「在最後一次修改之前都可以登入」
5. 意識到問題不在 Docker 映像，而在掛載的資料目錄

**根本原因**:
- `/root/taskflow-data/` 目錄中的文件被損壞
- 可能是 SQLite WAL 文件（`.db-shm`, `.db-wal`）損壞
- 容器崩潰時導致資料不一致

**關鍵發現**:
- ✅ 使用全新資料目錄 + 早期備份後立即可以登入
- ❌ Docker 映像回溯不會恢復損壞的資料目錄
- ❌ 資料庫 integrity_check 顯示 OK，但實際上有問題

### 問題 2：備份機制不完整

**現有備份的問題**:
1. **每日自動備份** (`/root/taskflow-backups/`)
   - ✅ 有定時備份
   - ❌ 只有資料庫，沒有 Docker 映像
   - ❌ 無法完整恢復系統

2. **容器內備份** (`/app/data/backups/`)
   - ✅ 可以手動觸發
   - ❌ 只有資料庫，沒有 Docker 映像
   - ❌ 容器刪除時可能丟失

3. **Docker 映像**
   - ✅ 有多個版本的映像
   - ❌ 沒有對應版本的資料庫備份
   - ❌ 映像和資料庫版本不匹配

**導致的問題**:
- 回溯 Docker 映像時，資料庫版本不匹配
- 使用舊的資料庫備份時，缺少新功能的資料表
- 無法找到「映像 + 資料庫」完全匹配的版本

### 問題 3：錯誤的診斷方向

**浪費時間的嘗試**:
1. ❌ 修復 `auth.js` exports 位置（問題不在這裡）
2. ❌ 檢查資料庫完整性（integrity_check 無法檢測 WAL 問題）
3. ❌ 驗證密碼哈希（密碼本身沒問題）
4. ❌ 修復 SQLite WAL 文件（已經損壞無法修復）
5. ❌ 多次回溯 Docker 映像（忽略了資料目錄問題）

**應該做的**:
1. ✅ 立即檢查資料目錄是否損壞
2. ✅ 使用全新資料目錄測試
3. ✅ 使用完整備份（映像 + 資料庫）恢復

## 🔧 解決方案

### 立即解決方法

1. **停止損壞的容器**
```bash
docker stop taskflow-pro
docker rm taskflow-pro
```

2. **備份損壞的資料目錄**
```bash
mv /root/taskflow-data /root/taskflow-data-broken
```

3. **創建全新資料目錄**
```bash
mkdir -p /root/taskflow-data-fresh
```

4. **恢復資料庫備份**
```bash
# 使用每日自動備份
cp /root/taskflow-backups/taskflow_backup_20260101_020001.db /root/taskflow-data-fresh/taskflow.db
cp /root/taskflow-data-broken/.db-key /root/taskflow-data-fresh/.db-key
```

5. **使用穩定映像啟動**
```bash
docker run -d --name taskflow-pro -p 3000:3000 \
  -v /root/taskflow-data-fresh:/app/data \
  taskflow-pro:v7.4.0-memos-fixed
```

6. **驗證功能**
```bash
docker logs taskflow-pro --tail 20
curl http://localhost:3000/api/health
```

## 🎓 關鍵教訓

### 1. Docker 映像回溯 ≠ 完全恢復

**錯誤認知**:
- ❌ 以為回溯 Docker 映像就能恢復系統
- ❌ 忽略了持久化資料在 volume 中
- ❌ 沒有意識到映像和資料庫版本必須匹配

**正確認知**:
- ✅ Docker 映像只包含應用程式代碼
- ✅ 持久化資料在掛載的 volume 中
- ✅ 回溯映像不會恢復損壞的資料目錄
- ✅ 必須同時恢復映像和資料庫

### 2. 資料目錄可能被操作損壞

**損壞原因**:
- 修改代碼時的錯誤導致容器崩潰
- SQLite WAL 文件在崩潰時損壞
- 文件權限問題
- 磁碟空間不足

**預防措施**:
- ✅ 修改前必須創建完整快照
- ✅ 測試時使用臨時資料目錄
- ✅ 定期清理 WAL 文件
- ✅ 監控磁碟空間

### 3. 只備份資料庫是不夠的

**問題**:
- 資料庫備份不包含應用程式代碼
- 映像和資料庫版本可能不匹配
- 缺少配置文件（.db-key 等）
- 無法完整恢復系統

**解決**:
- ✅ 創建完整系統快照（映像 + 資料庫 + 配置）
- ✅ 快照包含恢復說明（RESTORE.md）
- ✅ 定期測試恢復流程
- ✅ 保留多個穩定版本的快照

### 4. 診斷方向很重要

**錯誤方向**:
- 花太多時間檢查代碼
- 相信 integrity_check 的結果
- 忽略用戶提供的關鍵信息
- 沒有使用全新環境測試

**正確方向**:
- 聽取用戶反饋（"之前可以登入"）
- 使用排除法（全新環境測試）
- 檢查外部因素（資料目錄、權限）
- 快速驗證假設

### 5. 備份策略必須完整

**不完整的備份**:
- 只有資料庫，沒有映像
- 只有映像，沒有資料庫
- 映像和資料庫版本不對應
- 沒有配置文件

**完整的備份**:
- Docker 映像（應用程式代碼）
- 資料庫文件（用戶數據）
- 配置文件（.db-key 等）
- 恢復說明（RESTORE.md）
- 版本信息（image-info.txt）
